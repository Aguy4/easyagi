## 视频简介

演示采用**纯视觉方案**操作本地电脑的案例：Agent 打开桌面微信，监控指定联系人「通用智能」的消息，并在收到消息后自动回复（回复内容由模型自行决定）。

## 任务示例

> 你打开桌面上的微信，帮我看着通用智能发来的消息，一有消息你就负责回复他，回复的内容你自己定。

## 工作流架构

```
┌──────────────┐     桌面截图命令      ┌─────────────┐
│  大模型节点   │ ◄────────────────── │ 获取桌面状态 │
│ （操控电脑）  │                      └─────────────┘
└──────┬───────┘
       │ 接收整合后的桌面元素信息
       ▼
┌──────────────────────────────────────────┐
│  YOLO（OmniParser） + OCR（PaddleOCR）    │
│         ↓ 编程节点整合定位与文字           │
└──────────────────────────────────────────┘
```

### 核心思路

1. **截屏**获取当前桌面画面
2. **YOLO**：微软 OmniParser 模型，识别窗口可点击元素位置
3. **OCR**：百度 PaddleOCR，识别窗口内文字及位置
4. **编程节点**：合并 YOLO 与 OCR 结果 → 窗口元素坐标 + 文字信息
5. **回传大模型**：模型基于结构化桌面状态进行点击、输入等操作

### 为何不用大模型直接识图？

| 方式 | 问题 |
|------|------|
| 大模型直接识别截图 | 窗口定位不准、文字识别不准，鼠标点击易错位 |
| YOLO + OCR + 结构化融合 | 定位与文字分离处理，再汇总给模型，点击更可靠 |

## 技术栈

| 组件 | 选型 |
|------|------|
| 窗口元素检测 | Microsoft OmniParser（YOLO） |
| 文字识别 | 百度 PaddleOCR |
| 任务执行 | 大语言模型节点 |

## 适用场景

- 桌面 GUI 自动化（无官方 API 的客户端）
- RPA 类智能体
- 需要「看屏幕 → 理解 → 操作」的本地助手
